### 2.2 `exchange/mqttVanillaTransmission/callables/code.py`

**[HIGH] Timestamp format `sss` produces seconds, not milliseconds** — `code.py:98`

**File:** `PISync_2026-04-11_1427/ignition/script-python/exchange/mqttVanillaTransmission/callables/code.py`, line 98

**Impact.** The `"local"` timestamp in every MQTT payload that uses the `qualifiedValue` or `json` payload format carries a wrong sub-second component. A tag change at 14:56:13.025 is timestamped as `2024-07-30 14:56:13.013` (the seconds value `13` zero-padded to three digits) rather than `2024-07-30 14:56:13.025` (milliseconds `25` → `025`). Consumers relying on millisecond precision for deduplication, ordering, or PI time-series alignment receive corrupted timestamps.

**Explanation.** `system.date.format` delegates to Java's `java.text.SimpleDateFormat`. In that format language `s` means *second-of-minute* (0–59) and `S` means *millisecond* (0–999). The pattern `sss` repeats the second component three times with minimum width 3, producing `013` for second 13 — not milliseconds. The inline comment (`# 2024-07-30 14:56:13.025`) shows millisecond intent; only `SSS` (capital S) achieves it.

**Source.** Current code (defective):
```python
98          timestamp = system.date.format(tagQualifiedValue.timestamp,"yyyy-MM-dd HH:mm:ss.sss")
```

**Recommended Fix.**
```python
98          timestamp = system.date.format(tagQualifiedValue.timestamp,"yyyy-MM-dd HH:mm:ss.SSS")
```
